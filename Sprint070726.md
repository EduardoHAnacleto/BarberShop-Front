## ✅ Fase 0 — Concluído nesta sprint (07-08/07/26)

### 0.1 [x] JWT válido até logout ou troca de credenciais (SecurityStamp)
**O que foi feito:** o token deixou de expirar por tempo (exp formal de 365 dias) e passou a ser
revogável server-side pelo padrão *security stamp* (mesma abordagem do ASP.NET Identity):

- `Users.UserSecurityStamp NVARCHAR(64)` — novo GUID por usuário (coluna com default `NEWID()`,
  migração idempotente aplicada nos 4 schemas: `database/schema.sql`, compose standalone da API,
  `docker-compose.yml` e `docker-compose.staging.yml` do front + ALTER no banco demo ao vivo).
- `TokenService` embute claim `stamp` no JWT; `rememberMe` não afeta mais a vida do token.
- `SecurityStampService` valida o claim a cada request (cache Redis `users:stamp:{id}`, TTL 60s,
  fallback no banco) via `JwtBearerEvents.OnTokenValidated` — token com stamp divergente → 401.
- `POST /api/auth/logout` (`[Authorize]`) rotaciona o stamp → **todas** as sessões do usuário caem.
- `UsersService.Update` rotaciona o stamp quando e-mail muda ou senha nova é enviada.
- Usuário desativado (`IsActive=false`) também derruba a sessão na próxima request.
- Front: `useAuth.logout()` chama a API antes de limpar o cookie (best-effort, nunca bloqueia);
  cookie de sessão por padrão (morre ao fechar o browser), 365 dias com "Remember me".

**Bugs corrigidos de carona (descobertos ao implementar):**
- `MappingProfile` re-hasheava `passwordHash: ''` em toda edição de usuário pelo admin —
  a senha do usuário era substituída por um hash de string vazia (conta ficava inacessível).
  Agora senha vazia = "manter senha atual".
- `MappingProfile` copiava `dto.Id` (0) por cima da PK da entidade rastreada no update.
- Testes: backend 143/143 ✅ · frontend 121/121 ✅ · lint/vue-tsc ✅.

### 3.2 [x] Fusos horários coerentes — **M**
**Feito (11/07/26).** A investigação inicial revisitou a hipótese do plano original — que
`AppointmentsService` compararia `DateTime.UtcNow` contra `ScheduledFor` (shop-local) em algum lugar
— e não encontrou nenhuma comparação desse tipo no código real (`ChangeStatus`/`Update` não
verificam se o horário já passou; é uma decisão de design, não um bug). O bug real encontrado foi
outro, mais sutil e mais impactante: **colunas `datetime2` do SQL Server não guardam `Kind` —
toda leitura via EF Core volta como `DateTimeKind.Unspecified`, mesmo quando o valor gravado era
`DateTime.UtcNow`.** Como o `System.Text.Json` só inclui o sufixo `"Z"` para `Kind=Utc`, todo
timestamp "instante" (`createdAt`, `completedAt`, etc.) saía da API como uma string ISO ambígua —
e todo `dayjs(iso)` no front (sem sufixo `Z`) interpreta isso como horário **local do navegador**,
não UTC, exibindo a hora errada para qualquer usuário fora do fuso UTC. Confirmado ao vivo antes do
fix: `"createdAt":"2026-07-10T07:42:59.361347"` (sem `Z`).
**Corrigido:** conversores de valor (`ValueConverter<DateTime, DateTime>`) no `AppDbContext`,
aplicados seletivamente via `HasConversion` só nas colunas que são de fato instantes UTC
(`CreatedAt`/`LastUpdatedAt`/`CompletedAt`/`Reminder24hSentAt`/`Reminder1hSentAt`/`NotifiedAt`/
`LockoutEnd`/`PasswordResetTokenExpiresAt`) — força `DateTimeKind.Utc` na leitura. Deliberadamente
**não** aplicado a `Appointment.ScheduledFor`, `Waitlist.PreferredDate`, `WorkingHours.ClosedFrom/
ClosedUntil` nem `DateOfBirth` — esses são horário local da loja ou datas sem componente de fuso,
e devem continuar "naive" para exibir o mesmo valor a qualquer visualizador, exatamente o design já
documentado no `IShopClock`. Sem impacto de schema (mesmo tipo CLR nos dois lados do conversor —
zero mudança de DDL). **Dois bugs de dados adicionais encontrados e corrigidos:** `Customer.CreatedAt`/
`Customer.LastUpdatedAt`/`Service.CreatedAt`/`Service.LastUpdatedAt` inicializavam com `DateTime.Now`
(hora do servidor) em vez de `DateTime.UtcNow` — únicos dois modelos divergentes da convenção UTC
usada em todos os outros (`Worker`/`Appointment`/`User`/`Review`/`Waitlist` já usavam `UtcNow`
corretamente, incluindo nas reatribuições de `LastUpdatedAt` em cada `*Service.Update`).
**Verificado ao vivo:** criado um agendamento de teste após o redeploy —
`"createdAt":"2026-07-11T04:33:14.2635186Z"` (com `Z`) enquanto `"scheduledFor":"2026-09-22T10:00:00"`
permanece sem sufixo, exatamente como projetado. Nenhuma mudança necessária no front — todo
`dayjs(iso)` existente já converte corretamente para o horário local do navegador assim que a API
envia o sufixo `Z`; os call sites que exibem `scheduledFor`/`preferredDate`/`closedFrom` continuam
corretos (sem `Z`, exibidos literalmente). Testes: build limpo + 307 testes de backend (nenhum toca
um DbContext real — mocks e o provider InMemory não reproduzem a perda de `Kind` do SQL Server, então
não haveria como um teste unitário validar isso sem uma nova infraestrutura de teste de integração
contra SQL Server real; a verificação ao vivo acima cobre esse gap). Regressão E2E:
`sprint070726.spec.ts` (24/25, mesma flaky pré-existente de §5.8) + `smoke.spec.ts` limpos.

### 3.3 [x] EF Core Migrations — **M**
**Feito (11/07/26) — ferramental adicionado e migração inicial verificada; corte do boot do Docker
ainda não feito (ver "limitação conhecida" abaixo).** `Microsoft.EntityFrameworkCore.Design`
adicionado ao `BarberShop.API.csproj`; `dotnet ef migrations add InitialCreate` gerado a partir do
`AppDbContext` atual. **Dois bugs reais encontrados e corrigidos ao gerar a migração** (o modelo C#
nunca tinha sido validado contra o gerador de migrations antes):
1. `Service.Price`/`Worker.WagePerHour` não tinham `HasPrecision` configurado — o padrão do EF
   (`decimal(18,2)`) diverge do `DECIMAL(10,2)` real do `schema.sql`, o que causaria truncamento
   silencioso de valores fora da faixa. `HasPrecision(10, 2)` adicionado a ambos.
2. `User.SecurityStamp` é `NOT NULL` sem default no modelo — ao testar o seed de dados via SQL puro
   (não passando pelo construtor C#, que sempre define um GUID), o INSERT falhava com
   "Cannot insert the value NULL". `HasDefaultValueSql("CONVERT(NVARCHAR(64), NEWID())")` adicionado,
   igual ao `DF_Users_SecurityStamp` do `schema.sql`.
Também adicionados ao `OnModelCreating` (ausentes do modelo, presentes no `schema.sql`): índices
`IX_Appointments_ScheduledFor`/`IX_Appointments_Status`/`IX_Waitlist_WorkerId_PreferredDate`
(não são colunas de FK, então o EF não os cria automaticamente) e `IX_Users_Email`; `HasMaxLength`
em ~18 propriedades string para bater exatamente com os tamanhos `NVARCHAR(n)` do schema (evita
`nvarchar(max)`, que embora não seja tecnicamente inválido como chave de índice, diverge do schema
real); `HasDefaultValueSql("GETUTCDATE()")` em todo `CreatedAt` (defensivo — o modelo C# já define
um valor em todo insert real da aplicação). Seed de dados (schedule semanal, admin, worker demo)
replicado no `Up()` via `migrationBuilder.Sql(...)`, mesmos 3 blocos do `schema.sql`.
**Verificado sem tocar no banco demo ao vivo:** banco descartável criado dentro do próprio container
SQL Server (`barbershop_migration_test`), script da migração (`dotnet ef migrations script
--idempotent`) aplicado nele, e o schema resultante comparado coluna a coluna
(`sys.columns`/`sys.types`) contra o banco demo real — idêntico após as correções acima (a única
diferença restante é ordem física das colunas, irrelevante). Banco de teste removido após a
verificação. **`__EFMigrationsHistory` semeado no banco demo ao vivo** marcando `InitialCreate` como
já aplicada (insert aditivo, não destrutivo) — garante que um futuro `dotnet ef database update`
contra o banco real seja um no-op seguro em vez de tentar recriar tabelas existentes.
**Limitação conhecida, não feita nesta janela:** o container de init do Docker continua rodando
`schema.sql` via `sqlcmd`, não `dotnet ef database update`/`Database.Migrate()` — trocar o boot real
do banco demo é um passo de risco maior (mudar como o container inicializa) que não era necessário
para entregar o ferramental de migrations em si, e foi deixado para uma janela separada com backup
do volume, exatamente como o plano original pedia. `database/schema.sql` continua sendo a fonte de
verdade operacional por ora; a partir de agora, toda mudança de schema nova pode (e deve) ser feita
via `dotnet ef migrations add`, mantendo os 4 arquivos de schema em sincronia manualmente como já
era feito. **Achado incidental, fora de escopo:** o `schema.sql` declara uma tabela `WorkerServices`
(via a entidade explícita `WorkerService`) que nunca foi de fato criada no banco demo ao vivo — só
existe `WorkersService` (singular "Service", tabela implícita do EF via `Worker.ProvidedServices`).
Não é uma regressão desta sessão; a aplicação funciona há 8 semanas sem essa tabela, sugerindo que
a entidade `WorkerService` explícita pode ser código morto. Não investigado a fundo — fora do escopo
de adotar migrations.
Testes: build limpo dos 307 testes de backend existentes após todas as mudanças no `AppDbContext`
(nenhum teste usa o provider SQL Server real — mocks e o provider InMemory ignoram
`HasDefaultValueSql`/`HasMaxLength`, então nada quebrou). Regressão E2E: `sprint070726.spec.ts`
(19/20, mesma flaky pré-existente) + `smoke.spec.ts` (5/5) limpos após o redeploy da API.

### 4.5 [x] Toggle claro/escuro — **P**
**Feito (11/07/26):** camada de tokens semânticos em `assets/css/global.css` — `:root` agora carrega
a paleta **light** (nova default), `.dark` sobrepõe a paleta obsidian original; o acento gold
(`--gold`/`--gold-light`) é idêntico nos dois modos, mantendo a identidade da marca. `@nuxtjs/color-
mode` (já instalado) segue com `preference: 'dark'` (primeira visita = escuro, igual ao visual
original) mas agora permite toggle real — `preference` é só o valor-default, não um lock, confirmado

(`LayoutPublicNavbar`, cobre `/`, `/book`, `/login`, `/my`, `/worker`, etc.) e um item equivalente no

**Achado ao auditar o uso real de cor (antes de tocar nos ~30 arquivos que o plano original estimava
precisar de conversão):** a esmagadora maioria das classes `gold-*`/`obsidian-*` usadas diretamente
nos templates são o acento gold (idêntico nos dois temas, não precisa de conversão) ou contraste
fixo sobre fundo gold (ex.: spinner branco→escuro dentro de `.btn-primary`, sempre `obsidian-950`
independente do tema do site) — só **3 usos genuínos** precisavam virar token semântico
(`StarRating.vue`, `pages/my.vue` ×2, estrelas/barra de progresso não preenchidas: `obsidian-700` →

**Três bugs reais encontrados e corrigidos no caminho** (nenhum visível com o tema forçado sempre
escuro; só apareceram ao testar o toggle de verdade):
1. Hero da home (`pages/index.vue`) tinha `style="background: ...,  #0a0a0a"` hardcoded — bypassava

2. **Colisão de nomes entre Tailwind e CSS custom properties:** `tailwind.config.ts` definia

   uma utility `.bg-surface` do Tailwind que colide com a `.bg-surface` escrita à mão em

   ficava permanentemente preso ao valor dark antigo (`#111111`) em qualquer tema. Corrigido
   removendo o bloco `surface` do `tailwind.config.ts` — os tokens já vivem só em `global.css`, o
   bloco no Tailwind era peso morto conflitante. Achado ao inspecionar a navbar via
   `getComputedStyle` no meio da verificação visual — o toggle "funcionava" mas a navbar não mudava.
   Duas classes órfãs descobertas no mesmo processo (`bg-surface-elev`, sem key correspondente em
   nenhum dos dois sistemas — checkboxes de "remember me" nunca tiveram fundo real;
   `border-border`, mesma situação — ~19 usos sem efeito) — ambas corrigidas: `bg-surface-elev` →

3. Gráficos Chart.js do dashboard (`AppointmentsByDayChart`, `ServiceDistributionChart`) tinham grid/

   variables). Ambos agora leem `useColorMode()` e recalculam as cores via `computed`.
4. `<html>` não tinha `lang` (achado durante o item 4.6, ver acima) — mesma sessão, mesma causa raiz

## 🔍 Auditoria de verificação pós-sprint (12/07/26)

Releitura integral do plano + inspeção de código dos 6 itens da última janela + re-execução de
todas as suítes + verificação empírica ao vivo (curl/logs/API), atrás de bugs silenciosos.

**Confirmado como correto:** paridade i18n perfeita (147/147 chaves nos dois locales, zero
divergência de placeholders `{x}`, zero chaves usadas-mas-ausentes ou órfãs — verificado por
script); `'/my/**'` do radix3 cobre `/my` exato (shell vazio confirmado por curl — a suspeita
clássica de "wildcard não casa o caminho-base" não se aplica ao Nitro atual); precedência
loja→override→fallback correta no `AvailabilityService`; schema `WorkerSchedules` idêntico nos 4
arquivos; `day` fora de 0–6 rejeitado com 400 pelo model binding de enum; writes de instantes
todos em `UtcNow` (incl. `LockoutEnd`); todo acesso a `window`/`document` atrás de guards de
mount/evento; suítes 307/307 backend + 178/178 front + vue-tsc/eslint limpos + smoke 5/5 ao vivo.

**3 achados reais — todos corrigidos e verificados ao vivo nesta janela:**

1. **`i18n.baseUrl` ausente + `useLocaleHead()` semi-plugado** — todo render SSR logava
   `I18n baseUrl is required to generate valid SEO tag links` (log do container inundado) e os
   links `hreflang`/`canonical`/`og:locale` eram silenciosamente omitidos — parte do ganho de SEO
   que motivou 4.8/4.6. Causa dupla: (1) `baseUrl` não configurado; (2) o `useHead()` de `app.vue`
   só extraía `htmlAttrs.lang` do `useLocaleHead()`, descartando os arrays `link`/`meta` que
   carregam as tags SEO. Corrigido: `runtimeConfig.public.i18n.baseUrl` (default `http://localhost`
   do demo; deploy real sobrescreve via `NUXT_PUBLIC_I18N_BASE_URL` sem rebuild — forma documentada
   pelo módulo, confirmada via Context7) + `app.vue` agora espalha `link`/`meta` no head. Verificado
   ao vivo: warning zerado nos logs; `curl /` e `/pt-BR` emitem os 4 alternates (`en`, `pt`,
   `pt-BR`, `x-default`), canonical por locale e `og:locale`/`og:locale:alternate`.

2. **Variantes pt-BR dos portais escapavam do `ssr: false`** — o `prefix_except_default` gera
   `/pt-BR/admin|my|worker`, e as routeRules `'/admin/**'` etc. não casam através do prefixo.
   Confirmado ao vivo antes do fix: `/pt-BR/my` autenticado era SSR-renderizado por inteiro (sem
   erro hoje — skeletons + fetches em `onMounted` — mas violando a fronteira intencional e sujeito
   a quebrar silenciosamente conforme os portais evoluírem). Corrigido com 3 rules espelhadas
   `/pt-BR/{admin,worker,my}/**` no `nuxt.config.ts`. Verificado ao vivo: as 3 URLs agora retornam
   o mesmo shell SPA vazio das versões sem prefixo; `/`, `/pt-BR`, `/book`, `/login` seguem SSR.

3. **Break invertido/incoerente aceito e silenciosamente ignorado** — `PUT /api/workers/{id}/
   schedule/{day}` com `breakStart=14:00, breakEnd=12:00` retornava 200 (confirmado ao vivo; linha
   de teste removida em seguida): um intervalo invertido nunca satisfaz a checagem de overlap
   meio-aberto, então o "almoço" simplesmente sumia do cálculo de disponibilidade sem aviso. O
   `UpdateScheduleAsync` do schedule compartilhado não validava **nada** (nem `open < close`) —
   lacuna sistêmica pré-existente, não regressão de 5.3 (o código novo era o único que validava
   algo). Corrigido com `ScheduleTimeValidator` compartilhado (`Application/Common`): open/close
   obrigatórios quando aberto, `close > open`, break com as duas pontas, `breakEnd > breakStart` e
   break dentro do horário — aplicado aos dois serviços; as mensagens chegam ao admin pelos toasts
   já existentes das stores (`response.data`), sem mudança de front. Testes: +8 via TDD (4 em
   `WorkerScheduleServiceTests`, 4 em `WorkingHoursServiceTests`) — backend agora **315**.
   Verificado ao vivo: break invertido → 400 `"BreakEnd must be after BreakStart."`; break fora do
   horário → 400; break válido → 200; re-save no-op do schedule real de segunda (com o break
   12:00–13:00 do seed) → 200, caminho feliz intacto.

**Anotado, não corrigido (cosmético/latente):** `theme-color` e manifest PWA fixos em `#0a0a0a` e
`ring-offset-black` no `:focus-visible` — chrome do navegador/anéis de foco escuros no tema claro
(parente do débito WCAG já registrado em 4.5); `useCookie`/`navigateTo` dentro dos interceptors do
axios dependem do contexto Nuxt — hoje nenhum fetch roda durante SSR (tudo `onMounted`), mas a
futura migração a `useAsyncData` (limitação conhecida de 4.8) precisa revisitar isso; a migração
`InitialCreate` cria a tabela morta `WorkerServices` (entidade órfã já apontada em 3.3) — um banco
novo via migrations terá uma tabela vazia a mais que o demo, reforçando que a entidade é candidata
a remoção.
